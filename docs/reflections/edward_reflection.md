# Individual Reflections - Edward Chan

This project changed my relationship with testing. I came in thinking of E2E tests as a chore, something you write at the end to prove the thing works before you submit it. I leave thinking of them as the most valuable feedback mechanism I have access to during development.

The pattern held across both sprints. In Sprint 1, writing the auth and partner-linking E2E suite (Issue #6) forced me to exercise the full token lifecycle — creation, redemption, expiry, and self-redemption rejection — before any of those edge cases came up in manual testing. The self-redemption guard in the service layer passed every one of those checks on the first run, which felt like a confirmation that the upfront service-layer design had been correct. The clearest counter-example came in Sprint 2. I was writing Playwright tests for the couple profile route (Issue #21), and the tests kept failing because an unauthenticated user could reach the page. That wasn't a test problem, it was a real bug, a missing route guard. The tests didn't just verify the feature; they found a security gap that design review and code review had both missed. That reframing — tests as a continuous audit of assumptions and not a final checkbox — is something I'll carry into every project going forward.

Working with Claude Code reshaped how I think about code ownership. When I'm the one writing every line, I have full context on every decision. When AI is involved in some of the writing, the code is still mine to own and defend, but the process of reviewing AI-generated code taught me to read more carefully and question more deliberately. I found myself asking "why is this structured this way?" more often than I would have with my own code, which I tend to rationalize rather than question.

The security-reviewer agent was particularly instructive. Running a dedicated security pass on a PR before merging isn't something I had done systematically before, but having it configured as part of the workflow made it feel like a normal part of development rather than an extra step. After the Netlify/Render deployment went live (Issue #47), I ran a security pass on the CORS middleware and it flagged a case where `CLIENT_ORIGIN` falls back to localhost if unset — a silent misconfiguration that would leave a Render deploy open without any functional error to signal it. That's exactly the kind of thing that's easy to miss in review because it's not a functional bug, and it would have been invisible in the E2E suite.

The schema design work in Sprint 1 was where I felt most confident. Designing the `invite_tokens`, `profiles`, and `pairs` tables (Issue #8) required thinking through the full lifecycle of the data — how tokens expire, how linking creates a new `pairs` row, what happens on delink — before writing a line of application code. That upfront modeling paid off across the whole project: we never had to do a schema migration mid-sprint, which kept the development velocity higher than I expected. A smaller version of the same discipline showed up in Sprint 2 when we agreed before writing any UI that `pairs.location` overrides `profiles.location` on the discover card. Documenting that decision in the data model before touching the couple profile preview (Issue #20) meant neither of us had to reverse-engineer it from the code later.

The area I want to improve is feature ownership. I owned the backend schema and E2E tests in Sprint 1, and added the couple profile edit and preview pages (Issues #19 and #20) in Sprint 2 alongside the E2E suite (Issue #21). The Sprint 2 retro named the imbalance directly: Heather carried most of the feature work while I focused on E2E, and that limited my exposure to the frontend and service layer. In Sprint 3, the discovery feed E2E work (Issue #29) reinforced the pattern rather than breaking it. Going forward I want to work across the full stack in every sprint, even if it means owning smaller pieces, because the handoff friction between front and back is where the most interesting bugs live.

The most useful thing I built in this project might be the documentation. Writing the schema decisions down in CLAUDE.md, articulating the state machine, defining what a "complete couple" means that work made the codebase legible to my partner and to my future self in a way that comments inside files never quite achieve.
