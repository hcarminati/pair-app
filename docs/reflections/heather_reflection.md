# Individual Reflections - Heather Carminati

Building Pair was the first time I used AI assistance, specifically Claude Code, not just as a code generator but as an active participant in a software development workflow and the difference in how I think about that tool has been the most meaningful outcome of this project.

Before taking this course, I treated Claude Code the way I had used autocomplete: prompt it, accept or reject, move on. The first time I saw a real shift was when I set up the `add-feature` skill. Writing a skill forces you to articulate your own conventions explicitly, like what does "done" mean for a feature in this codebase? What checks need to pass? What tests need to exist first? I found myself writing things down that I had previously held loosely in my head, and the act of codifying them made me a more consistent developer, independent of whether Claude followed them.

The TDD workflow was the other inflection point, but I have to be honest that it took longer than I expected to actually implement it. The project committed to test-first from Sprint 1, and I understood the rationale, but in the early sprints I was writing tests after the fact and telling myself they counted. It wasn't until Sprint 6 that I implemented TDD properly. For the connections tab (Issue #43), I committed the UI tests first, watched CI fail, then built the component to make them pass. For the chat feature (Issue #44), the test-first order covered all three layers: tests committed, then server functionality, then UI, each iteration failing CI until the implementation caught up. In both cases, running the `/pipeline` skill after implementation caught the remaining errors cleanly before pushing.

What the Sprint 6 experience made clear was how much the failing test phase functions as a specification step. Writing a test for the chat thread before any backend code exists forces you to decide what the message shape looks like, how sender identity is represented, and what the loading state should be. Those decisions made the implementation straightforward in a way that earlier features, where I reached for the keyboard before the tests did not. The red-green-refactor loop isn't just a process ritual; it's a thinking tool, and I should have trusted it earlier.

What surprised me about Claude Code specifically was the hooks system. The Stop hook that blocks session exit if tests are failing has saved me from committing broken code more than once. The cognitive load of switching contexts mid-sprint is real, and having an automated check that doesn't rely on my remembering to run tests is genuinely valuable. The PostToolUse Prettier hook is smaller but similarly impactful: formatting is no longer a decision I make, so I never have a formatting debate with myself or with a partner.

The hardest part of this project was the connection state machine. The logic (dual consent across four participants, with a state that can be vetoed at any stage by anyone) is inherently complex, and keeping that logic in the Express service layer rather than leaking it into route handlers or the frontend required discipline every time I touched a related endpoint. This became especially clear in Sprint 4 when I was wiring up the chat thread, the UI needed to know whether a connection was in the CONNECTED state before opening a Realtime subscription, and the temptation to check that state on the client was real.
Deploying in Sprint 3 was its own lesson. We treated deployment as a feature to be scheduled, not an afterthought, but we still ran into three CORS issues in quick succession after the Netlify frontend went live against the Render backend. The fix commits are in the git history and they were avoidable. A deployment smoke-test checklist run after every production deploy would have caught them in one pass. 

If I were starting over, I would invest earlier in the MCP server integration. Having GitHub Issues, PR comments, and branch management available inside the Claude Code session reduced context-switching significantly once it was set up, but I set it up later than I should have. That tooling changes what questions you can ask and answer without leaving your editor.
